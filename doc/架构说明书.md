## 1. 项目概述

**Visual Rhythm** 是一款基于微信小游戏平台开发的下落式节奏游戏。玩家需要在音符（Note）下落到判定线时，按下对应的虚拟按键（A/S/D/F），根据时机准确度获得 PERFECT / GOOD / MISS 三种判定，并积累分数和连击数。

### 核心玩法

- 屏幕分为 4 条垂直轨道，分别对应 A、S、D、F 按键
- 节奏点（Note）从屏幕顶部沿轨道匀速下落
- 判定线位于屏幕 2/3 处，上下各有 PERFECT 和 GOOD 判定区域
- 支持多种难度的预设节奏模式（从 JSON 配置文件加载）
- 游戏内可通过设置按钮切换节奏模式

## 4. 整体架构设计

项目采用**分层 MVC 变体架构**，结合**事件驱动**和**单向数据流**的设计思想：

```
┌─────────────────────────────────────────────────────────────┐
│                        入口层                                │
│   game.js → new Main()                                      │
└────────────────────────┬────────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────────┐
│                    主控制器层（Main）                         │
│  游戏循环（loop）/ 状态调度 / 输入路由 / 判定逻辑              │
└──┬──────────────────────────────────────────────────────────┘
   │
   ├─────────────────────────────────────────────────────────┐
   │                   节奏子系统（rhythm/）                   │
   │  TrackSystem / Note / BeatGenerator /                   │
   │  PatternLoader / RhythmSelector / VirtualKeys           │
   └─────────────────────────────────────────────────────────┘
   │
   ├─────────────────────────────────────────────────────────┐
   │                  运行时组件（runtime/）                   │
   │  BackGround / GameInfo / Music                          │
   └─────────────────────────────────────────────────────────┘
   │
   ├─────────────────────────────────────────────────────────┐
   │                 全局状态层（DataBus）                     │
   │  分数 / 连击 / BPM / Notes 列表 / 动画列表 / 对象池        │
   └─────────────────────────────────────────────────────────┘
   │
   └─────────────────────────────────────────────────────────┐
                       基础工具层（base/ & libs/）              │
       Sprite / Animation / Pool / TinyEmitter               │
       └─────────────────────────────────────────────────────┘
```

### 架构关键原则

1. **单向数据流**：游戏状态统一存储在 `DataBus`（单例），各组件读取状态，由 `Main` 统一更新。
2. **事件解耦**：组件间通过 `TinyEmitter` 发布/订阅事件（如游戏重启），避免直接引用。
3. **全局命名空间**：`GameGlobal.databus`、`GameGlobal.musicManager`、`GameGlobal.main` 在微信小游戏的 `GameGlobal` 全局对象上挂载，供各模块访问。
4. **对象池复用**：`Note` 对象通过 `Pool` 类复用，减少垃圾回收开销。

### 5.1 入口层

**文件**：`game.js`

```js
import Main from './js/main';
new Main();
```

职责单一：仅负责实例化主控制器 `Main`，启动整个游戏。

---

### 5.2 主控制器（Main）

**文件**：`js/main.js`

`Main` 是游戏的核心调度中心，承担以下职责：

| 职责 | 说明 |
|------|------|
| **初始化全局资源** | 创建 `DataBus`、`Music`，并挂载到 `GameGlobal` |
| **管理游戏状态** | 维护 `gameStarted` 标志，控制选择器/游戏切换 |
| **驱动游戏循环** | `loop()` → `update()` + `render()`，通过 `requestAnimationFrame` 调度 |
| **输入路由** | 将触摸事件路由到模式按钮或虚拟按键处理 |
| **判定逻辑** | `checkNoteHit()` 检查命中，`checkMissedNotes()` 检测 MISS |
| **状态重置** | `start()`、`restart()`、`resetGameState()` 三种重置粒度 |

#### 游戏循环

```
requestAnimationFrame
       │
       ▼
    loop()
    ├── update()
    │     ├── frame++
    │     ├── bg.update()
    │     ├── beatGenerator.generateBeats()   // 生成节奏点
    │     ├── notes.forEach(note.update())    // 更新节奏点位置
    │     ├── checkMissedNotes()              // 检测 MISS
    │     └── updateKeyHighlights()           // 更新高亮效果
    └── render()
          ├── ctx.clearRect()
          ├── bg.render()
          ├── trackSystem.render()            // 绘制轨道和判定区
          ├── notes.forEach(note.render())    // 绘制节奏点
          ├── renderKeyHighlights()
          ├── virtualKeys.render()
          ├── gameInfo.render()
          └── animations.forEach(ani.aniRender())
```

### 5.3 全局状态管理（DataBus）

**文件**：`js/databus.js`

`DataBus` 实现**单例模式**，是整个游戏的唯一状态容器。

#### 核心状态字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `notes` | `Note[]` | 当前活跃的节奏点列表 |
| `animations` | `Animation[]` | 当前活跃的动画列表 |
| `frame` | `number` | 当前帧编号（自游戏开始递增） |
| `score` | `number` | 当前总分数 |
| `combo` | `number` | 当前连击数 |
| `maxCombo` | `number` | 本局最大连击数 |
| `bpm` | `number` | 当前 BPM（每分钟节拍数） |
| `isGameOver` | `boolean` | 游戏是否已结束 |
| `perfectHits` | `number` | PERFECT 判定次数 |
| `goodHits` | `number` | GOOD 判定次数 |
| `missHits` | `number` | MISS 判定次数 |
| `pool` | `Pool` | 对象池实例（复用 Note） |
| `onKeyPress` | `Function` | 按键事件回调（由 Main 注入） |

#### 评分机制

```
PERFECT 命中：基础分 300 × 连击加成（最高 3×）
GOOD    命中：基础分 200 × 连击加成（最高 3×）
MISS        ：连击归零，不加分

连击加成倍率 = Math.min(1 + combo × 0.1, 3)
精度 = (perfectHits + goodHits) / totalHits × 100%
```

### 5.4 节奏子系统（rhythm/）

#### 5.4.1 轨道系统（TrackSystem）

**文件**：`js/rhythm/track.js`

管理 4 条垂直轨道的渲染和判定区域计算。

```
屏幕布局
┌──────┬──────┬──────┬──────┐
│  轨0 │  轨1 │  轨2 │  轨3 │  ← 4 条等宽轨道（颜色各异）
│  (A) │  (S) │  (D) │  (F) │
│      │      │      │      │
│      │      │      │      │
├──────┴──────┴──────┴──────┤ ← 判定区域上边界（屏幕 1/2 处）
│       GOOD 区（绿色，低透）  │
├──────────────────────────┤ ← PERFECT 区上边界（屏幕 7/12 处）
│     PERFECT 区（金色，中透）  │
├──────────────────────────┤ ← 判定线（屏幕 2/3 处，白色实线）
│     PERFECT 区下半部分      │
├──────────────────────────┤ ← PERFECT 区下边界
│       GOOD 区（绿色，低透）  │
├──────────────────────────┤ ← 判定区域下边界（屏幕 5/6 处）
│      虚拟按键触控区（透明）   │
└──────────────────────────┘
```

| 区域 | 位置 | 判定结果 |
|------|------|---------|
| PERFECT 区 | 判定线上下各 1/12 屏幕高度 | PERFECT |
| GOOD 区 | 判定区域中除 PERFECT 外的部分 | GOOD |
| 判定区域外 | 超出后 50px | MISS |

#### 5.4.2 节奏点（Note）

**文件**：`js/rhythm/note.js`

继承 `Sprite`，代表一个从顶部下落的节拍元素。

| 属性 | 说明 |
|------|------|
| `track` | 所在轨道索引（0-3） |
| `speed` | 下落速度（4 px/帧） |
| `isActive` | 是否活跃（false 则不渲染/更新） |
| `isStrongNote` | 强音符（金色边框，更亮颜色） |
| `isWeakNote` | 弱音符（灰色边框，较暗颜色） |
| `is8thNote` | 8 分音符（红色边框，显示数字"8"） |
| `is16thNote` | 16 分音符（青色边框，显示数字"16"） |

生命周期：`init()` → `update()` → `destroy()` → 回收进对象池

#### 5.4.3 节拍生成器（BeatGenerator）

**文件**：`js/rhythm/beatgenerator.js`

基于当前时间和 JSON 节奏模式，按节拍时序生成 Note 对象。

**工作流程**：

```
generateBeats(now)
    │
    ├── 第一次调用：记录 patternStartTime，解析 currentNotes 序列
    │
    ├── 计算 elapsedTime = now - patternStartTime
    ├── 计算 currentTimeInPattern = elapsedTime % patternDuration
    │
    ├── 检测循环重置（currentTimeInPattern < lastPatternTime）
    │     └── 重置所有 note.generated = false
    │
    └── 遍历 currentNotes，对满足以下条件的音符生成 Note：
          !note.generated && currentTimeInPattern >= note.timing
```

**时间计算**：
- 每拍时间 = `60000 / bpm` 毫秒
- 每单位时间 = `每拍时间 / 2`（每个字符代表一个 8 分音符）
- 模式总时长 = `最长轨道字符数 × 每单位时间`

#### 5.4.4 节奏模式加载器（PatternLoader）

**文件**：`js/rhythm/patternloader.js`

负责将 JSON 格式的节奏模式数据解析为可供 `BeatGenerator` 使用的音符序列。

**解析流程**：

```
JSON 文件（rhythm-patterns.json）
         │
         ▼
  loadPatterns(patternData)
         │
         ▼
  setCurrentPattern(patternId)
         │
         ▼
  getCurrentPatternNotes()
         │
         ▼
  parseTrackPattern(pattern)
    ├── 遍历每条轨道字符串
    └── parseTrackString(trackStr, trackIndex)
          ├── 'X' → 强音符（velocity=1.0）
          ├── 'x' → 弱音符（velocity=0.7）
          └── '-' → 休止符（跳过）
```

支持功能：
- 按难度筛选模式（`getPatternsByDifficulty`）
- BPM 比例缩放音符时间（`adjustNotesBPM`）
- 获取可视化配置（轨道颜色、符号说明）

#### 5.4.5 节奏选择器（RhythmSelector）

**文件**：`js/rhythm/rhythmselector.js`

游戏开始前和游戏中唤醒时显示的模式选择 UI 组件，实现纯 Canvas 绘制。

支持两种显示模式：
- **`initial`**：游戏开始前全屏显示
- **`in-game`**：游戏进行中叠加显示（覆盖最上层）

核心交互：
- 下拉菜单选择节奏模式
- 双击 BPM 输入框进入虚拟键盘输入模式
- 点击"开始游戏"触发 `onStartGame` 回调

#### 5.4.6 虚拟按键（VirtualKeys）

**文件**：`js/rhythm/virtualkeys.js`

管理屏幕下方的 A/S/D/F 四个透明触控区域，并统一处理触摸和键盘两种输入源。

- 触控区域占据屏幕底部 1/3 高度
- 按键视觉完全透明（仅保留触控功能）
- 按键按下时触发震动反馈（`wx.vibrateShort`）
- 通过 `DataBus.handleKeyPress()` 将事件传递给主控制器

### 5.5 运行时子系统（runtime/）

#### 5.5.1 背景（BackGround）

**文件**：`js/runtime/background.js`

继承 `Sprite`，实现无限滚动背景（双图片循环）：

- 每帧 `top += 2`（速度 2 px/帧）
- 到达屏幕高度时重置为 0
- 同时绘制两张背景图，无缝衔接

#### 5.5.2 游戏 HUD（GameInfo）

**文件**：`js/runtime/gameinfo.js`

继承 `TinyEmitter`，负责运行时 UI 展示：

- **运行时 HUD**：左上角显示分数、连击、BPM；右上角显示精度、最大连击、P/G/M 统计
- **结算界面**：游戏结束时显示结果面板（最终得分、最大连击、精度、详细统计）
- **重启按钮**：点击触发 `'restart'` 事件，由 `Main` 监听处理

#### 5.5.3 音频管理（Music）

**文件**：`js/runtime/music.js`

实现**双轨音频策略**（主路径 + 降级路径）：

| 优先级 | 方式 | 说明 |
|--------|------|------|
| 1（主） | 微信 `InnerAudioContext` | 播放 MP3 文件（bgm/bullet/boom） |
| 2（降级） | Web Audio API | 程序生成鼓声（指数衰减包络+噪声） |

音效映射：
```
PERFECT → playDrumHit() + playPerfect()   (bullet.mp3)
GOOD    → playDrumHit() + playGood()      (bullet.mp3)
MISS    → playMiss()                       (boom.mp3)
```

### 5.6 基础工具层（base/ & libs/）

#### Sprite（`js/base/sprite.js`）

所有游戏实体的基类，继承自 `TinyEmitter`：

| 能力 | 说明 |
|------|------|
| 图片加载 | `wx.createImage()` + `img.src` |
| 位置管理 | `x`, `y`, `width`, `height` |
| 可见性 | `visible` 控制渲染 |
| 碰撞检测 | `isCollideWith(sp)` 中心点包含检测 |
| 渲染 | `render(ctx)` 绘制图片 |

#### Animation（`js/base/animation.js`）

继承 `Sprite`，提供帧动画能力：

- `initFrames(imgList)` 加载帧序列并注册到 `DataBus.animations`
- `playAnimation(index, loop)` 启动帧定时器（`setInterval`）
- `stopAnimation()` 停止并发射 `'stopAnimation'` 事件

#### Pool（`js/base/pool.js`）

对象池实现，减少 `Note` 对象的频繁创建和垃圾回收：

```js
// 获取（优先从池中取，池空则 new）
pool.getItemByClass('note', Note)

// 回收（Note.destroy() 时调用）
pool.recover('note', note)
```

#### TinyEmitter（`js/libs/tinyemitter.js`）

轻量级事件发射器，提供 `on`、`once`、`emit`、`off` 接口，被 `Sprite`、`GameInfo` 等继承使用。

## 6. 设计模式应用

| 设计模式 | 应用位置 | 说明 |
|---------|---------|------|
| **单例模式** | `DataBus`、`Music` | 通过 `instance` 变量保证全局唯一实例 |
| **对象池模式** | `Pool` + `Note` | 复用 Note 对象，避免频繁 GC |
| **观察者模式** | `TinyEmitter` | `GameInfo` 发布 `'restart'`，`Main` 订阅 |
| **策略模式** | `PatternLoader` | 支持不同节奏模式的解析策略 |
| **模板方法模式** | `Sprite → Note/BackGround` | 基类定义 `render/update` 模板，子类重写 |
| **外观模式** | `DataBus` | 为游戏状态提供统一访问接口 |
| **回调注入模式** | `RhythmSelector.onStartGame` | 通过属性注入回调，降低耦合 |

## 8. 渲染管线

每帧渲染按以下顺序（从后到前，遵循画家算法）：

```
1. ctx.clearRect()           // 清空画布
2. bg.render()               // 背景（最底层）
   ↓ [游戏未开始时]
3a. rhythmSelector.render()  // 节奏选择器（退出渲染）
   ↓ [游戏进行中]
3b. renderModeButton()       // 模式唤醒按钮（左上角齿轮图标）
4.  trackSystem.render()     // 轨道背景 + 判定区 + 判定线 + 按键提示
5.  notes.forEach(render)    // 所有活跃节奏点
6.  renderKeyHighlights()    // 按键高亮效果（半透明白色覆盖）
7.  virtualKeys.render()     // 虚拟按键（当前版本透明，不绘制）
8.  gameInfo.render()        // HUD（分数/连击/BPM）+ 结算界面
9.  animations.forEach()     // 帧动画（爆炸等特效，最顶层）
   ↓ [游戏中选择器被唤醒时]
10. rhythmSelector.render()  // 在游戏中叠加显示的选择器（最顶层）
```

## 13. 性能优化策略

### 对象池

`Note` 对象通过 `Pool` 复用，生命周期：

```
new Note() → init() → [游戏中活跃] → destroy() → pool.recover() → [下次复用]
```

### 渲染优化

- 每帧清空画布后全量重绘（适合小游戏简单场景）
- 按键高亮效果使用帧计数器（`keyHighlightTimers`），避免 setTimeout

### MISS 检测节流

`checkMissedNotes()` 在每帧 `update()` 中调用，但 Note 超出判定区 50px 容错距离，避免边界误判。

### 调试日志节流

非关键调试信息使用 `frame % 60 === 0` 条件过滤，每秒只打印一次。
