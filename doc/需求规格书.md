## 1. 引言

### 1.1 目的

本文档是"视觉节奏游戏"（Visual Rhythm Game）的软件需求规格书，描述该系统的功能性需求与非功能性需求，供开发人员、测试人员及相关干系人使用。

### 1.2 范围

本游戏基于微信小程序平台开发，是一款音乐节奏类游戏。玩家需要在节奏点（Note）下落至判定区域时按下对应按键，通过精确的时机操作获得分数。游戏支持多种节奏模式与难度等级，提供多维度的即时反馈（视觉、音效、触觉）。

本文档涵盖的主要功能模块：

- 游戏主循环与渲染
- 轨道与节奏点系统
- 虚拟按键与输入处理
- 节拍生成与节奏模式管理
- 判定系统与评分系统
- 音效管理
- 节奏选择器（游戏配置界面）
- 全局状态管理（DataBus）

### 1.3 定义、缩略词与缩写

| 术语 | 说明 |
|------|------|
| Note / 节奏点 | 从屏幕顶部落下的可被玩家击打的节拍元素 |
| BPM | Beats Per Minute，每分钟节拍数，表示音乐速度 |
| PERFECT | 判定等级：节奏点中心位于PERFECT判定区域内时的命中结果 |
| GOOD | 判定等级：节奏点中心位于GOOD判定区域内时的命中结果 |
| MISS | 判定等级：节奏点未被命中或超出判定区域时的结果 |
| 判定线 | 屏幕高度2/3处的白色横线，玩家应在此处击打节奏点 |
| 判定区域 | 以判定线为中心、高度为屏幕高度1/3的区域 |
| 连击 | Combo，连续成功命中（PERFECT或GOOD）的次数 |
| DataBus | 全局状态管理单例，负责管理游戏运行时数据 |
| 节奏模式 | 预定义的节拍序列，由JSON配置文件描述 |
| 对象池 | Pool，用于复用游戏对象、优化内存分配的数据结构 |
| 轨道 | Track，屏幕垂直分割的四个区域之一，对应一个操作按键 |
| FPS | Frames Per Second，每秒帧数 |
| Web Audio API | 浏览器/小程序环境下的音频处理接口 |
| Canvas 2D | 二维图形渲染接口，用于绘制游戏画面 |

## 2. 综合描述

### 2.1 产品概述

视觉节奏游戏是一款运行于微信小程序平台的音乐节奏类游戏，采用竖屏布局。游戏界面包含四条垂直轨道，节奏点从屏幕顶部以固定速度下落，玩家需要在节奏点到达判定区域时按下对应的虚拟按键（或键盘按键），以获得分数。

游戏的核心体验要素：
- **准确性挑战**：玩家需要在正确的时机操作，判定精度决定分数等级
- **节奏感训练**：多样的节奏模式帮助玩家感受不同风格的音乐节拍
- **成就激励**：连击系统和分数加成机制持续激励玩家提升表现

### 2.2 产品功能

本游戏主要功能概览：

| 功能类别 | 子功能 |
|----------|--------|
| 游戏主循环 | 帧渲染、逻辑更新、动画管理 |
| 节奏点系统 | 生成、下落、渲染、判定、回收 |
| 轨道系统 | 四轨道布局、判定区可视化、按键高亮 |
| 输入处理 | 触摸事件、键盘事件、防重复输入 |
| 节拍生成 | 基于JSON模式的节拍序列、BPM动态管理 |
| 判定系统 | PERFECT/GOOD/MISS三级判定 |
| 评分系统 | 基础分数、连击加成、精度统计 |
| 音效管理 | 背景音乐、命中音效、Web Audio合成 |
| 触觉反馈 | 按键震动、命中震动（微信小程序接口） |
| 节奏选择器 | 模式选择、BPM设置、虚拟键盘 |
| 游戏状态管理 | 全局状态、游戏开始/结束/重置 |

### 2.3 用户特性

| 用户类型 | 特征描述 |
|----------|----------|
| 普通玩家 | 无需专业音乐知识，通过直观操作即可上手 |
| 进阶玩家 | 追求高分和高连击，对节奏精度有较高要求 |
| 音乐爱好者 | 关注节奏模式的多样性和音乐感 |

### 2.4 运行环境

- **主要环境**：微信小程序（基于 Canvas 2D 渲染）
- **兼容环境**：支持现代浏览器（用于开发调试）
- **屏幕方向**：竖屏（Portrait）

### 2.5 假设与依赖

1. 微信小程序 API 可用（`wx.onTouchStart`、`wx.vibrateShort`等）
2. 设备支持 Canvas 2D 渲染
3. 设备支持 Web Audio API（用于音效合成），若不支持则降级处理
4. 节奏模式配置文件（`rhythm-patterns.json`）位于小程序根目录，且格式合法
5. 音频资源（`audio/bgm.mp3`、`audio/bullet.mp3`、`audio/boom.mp3`）位于对应目录，若文件缺失则降级为静音模式

## 3. 功能需求

### 3.1 游戏主循环

#### 3.1.1 帧循环

**需求编号**：FR-LOOP-001  
**描述**：系统应以 60 FPS 的目标帧率持续运行游戏主循环，每帧执行逻辑更新（update）和画面渲染（render）。  
**实现参考**：`js/main.js` → `loop()`

#### 3.1.2 游戏状态控制

**需求编号**：FR-LOOP-002  
**描述**：系统应维护游戏状态（`gameStarted`、`isGameOver`），并根据状态决定是否执行节奏点生成、逻辑更新等操作。  
**状态说明**：
- 未开始：显示节奏选择器，不生成节奏点
- 游戏中：正常运行主循环
- 游戏结束：停止逻辑更新，显示结算界面

#### 3.1.3 全局对象初始化

**需求编号**：FR-LOOP-003  
**描述**：游戏启动时，系统应初始化全局单例对象 `DataBus`（数据管理）和 `Music`（音效管理），并将 `Main` 实例挂载到 `GameGlobal.main`，供各组件访问。

---

### 3.2 轨道系统

#### 3.2.1 四轨道布局

**需求编号**：FR-TRACK-001  
**描述**：游戏界面应在水平方向均等划分为四条垂直轨道，每条轨道宽度为屏幕宽度的1/4。  
**轨道颜色（半透明背景）**：
| 轨道索引 | 颜色 | 对应按键 |
|----------|------|----------|
| 0 | `#FF6B6B`（红） | A |
| 1 | `#4ECDC4`（青） | S |
| 2 | `#45B7D1`（蓝） | D |
| 3 | `#96CEB4`（绿） | F |

#### 3.2.2 判定线与判定区域

**需求编号**：FR-TRACK-002  
**描述**：系统应在屏幕高度2/3处绘制白色判定线（线宽3px），并渲染以下判定区域：
- **总判定区域**：以判定线为中心，高度为屏幕高度的1/3
- **PERFECT区域**：以判定线为中心，高度为屏幕高度的1/6，使用金色（`#FFD700`）半透明填充
- **GOOD区域**：总判定区域中除PERFECT区域外的上下两段，使用绿色（`#00FF00`）低透明度填充

#### 3.2.3 按键提示

**需求编号**：FR-TRACK-003  
**描述**：系统应在判定线上方各轨道中央显示对应的按键字母（A/S/D/F），使用白色粗体字体（`bold 24px Arial`）。

#### 3.2.4 轨道高亮反馈

**需求编号**：FR-TRACK-004  
**描述**：当玩家按下某轨道对应按键时，该轨道的判定区域应立即显示白色半透明高亮效果，持续10帧后消失。

---

### 3.3 节奏点（Note）系统

#### 3.3.1 节奏点属性

**需求编号**：FR-NOTE-001  
**描述**：每个节奏点（Note）应具备以下属性：
| 属性 | 值 | 说明 |
|------|-----|------|
| 宽度 | 60px | 固定宽度 |
| 高度 | 20px | 固定高度 |
| 下落速度 | 4px/帧 | 固定速度 |
| 轨道 | 0~3 | 所在轨道索引 |
| 颜色 | 见3.2.1 | 根据轨道决定 |
| 音符类型 | 普通/强/弱/8分/16分 | 影响外观 |

#### 3.3.2 节奏点生成位置

**需求编号**：FR-NOTE-002  
**描述**：节奏点在对应轨道水平居中，初始Y坐标为 `-height`（即屏幕顶部以上），从该位置开始向下下落。

#### 3.3.3 节奏点渲染规则

**需求编号**：FR-NOTE-003  
**描述**：节奏点的外观应根据类型进行区分：
| 类型 | 边框颜色 | 边框宽度 | 填充色处理 |
|------|----------|----------|------------|
| 普通 | `#FFFFFF` | 2px | 默认轨道颜色 |
| 强音符 | `#FFD700`（金） | 3px | 亮度+30% |
| 弱音符 | `#CCCCCC`（灰） | 1px | 亮度-30% |
| 8分音符 | `#FF6B6B` | 2px | 默认，中央显示"8" |
| 16分音符 | `#4ECDC4` | 2px | 默认，中央显示"16" |

#### 3.3.4 节奏点生命周期管理

**需求编号**：FR-NOTE-004  
**描述**：节奏点应使用对象池（Pool）管理，在以下情况下被回收：
- 玩家成功命中（调用 `destroy()`）
- 节奏点超出屏幕底部（`y > SCREEN_HEIGHT + 50`），触发MISS判定后回收

### 3.4 输入处理

#### 3.4.1 触摸事件

**需求编号**：FR-INPUT-001  
**描述**：系统应监听 `wx.onTouchStart` 事件，支持多点触控。当触摸点Y坐标位于虚拟按键区域（屏幕下方1/3区域）时，根据X坐标判断所属轨道，触发对应轨道的按键动作。

#### 3.4.2 键盘事件

**需求编号**：FR-INPUT-002  
**描述**：在支持 `wx.onKeyDown` 的环境（如开发者工具），系统应监听键盘事件：
| 键位 | 轨道 |
|------|------|
| A | 0（第一轨道） |
| S | 1（第二轨道） |
| D | 2（第三轨道） |
| F | 3（第四轨道） |
同一按键在上次弹起前不重复触发（防重复输入）。

#### 3.4.3 游戏结束时输入屏蔽

**需求编号**：FR-INPUT-003  
**描述**：当游戏处于结束状态（`isGameOver === true`）时，系统应忽略所有轨道按键输入。

#### 3.4.4 模式唤醒按钮

**需求编号**：FR-INPUT-004  
**描述**：游戏进行中，屏幕左上角应显示一个60×60px的齿轮图标按钮（左上边距均为20px）。点击该按钮时，系统应叠加显示节奏选择器（in-game模式），同时触发轻微震动反馈。

---

### 3.5 节拍生成

#### 3.5.1 基于JSON模式的节拍生成

**需求编号**：FR-BEAT-001  
**描述**：系统应从 `rhythm-patterns.json` 加载节奏模式库，根据玩家选择的模式，按照轨道定义字符串按时序生成节奏点。模式字符串规则：
| 字符 | 含义 |
|------|------|
| `X` | 强音符，在该时刻生成一个节奏点 |
| `x` | 弱音符，在该时刻生成一个节奏点 |
| `-` | 休止符，不生成节奏点 |

#### 3.5.2 节拍时序计算

**需求编号**：FR-BEAT-002  
**描述**：每个字符的持续时间为半拍（即 `(60/BPM×1000)/2` 毫秒）。系统应基于当前时间与模式开始时间，计算出当前时刻在模式中的位置，按时生成对应的节奏点，节奏模式循环播放。

#### 3.5.3 BPM更新

**需求编号**：FR-BEAT-003  
**描述**：BPM可在游戏配置界面由玩家设置（范围40~200），设置后同步更新 `BeatGenerator` 和 `DataBus` 中的BPM值。

#### 3.5.4 降级策略

**需求编号**：FR-BEAT-004  
**描述**：若 `rhythm-patterns.json` 加载失败，系统应自动使用内置的默认节奏模式（`basic` 和 `eighth`），保证游戏可正常运行。

### 3.6 判定系统

#### 3.6.1 PERFECT判定

**需求编号**：FR-JUDGE-001  
**描述**：当玩家按键时，若对应轨道上在判定区域内的最近节奏点中心（`y + height/2`）位于PERFECT区域内（以判定线为中心，上下各1/12屏幕高度），判定结果为 `PERFECT`。

#### 3.6.2 GOOD判定

**需求编号**：FR-JUDGE-002  
**描述**：当玩家按键时，若最近节奏点中心位于总判定区域内但不在PERFECT区域内，判定结果为 `GOOD`。

#### 3.6.3 MISS判定（主动）

**需求编号**：FR-JUDGE-003  
**描述**：当玩家按键时，若对应轨道判定区域内无活跃节奏点，不产生任何判定结果（不触发MISS）。

#### 3.6.4 MISS判定（被动）

**需求编号**：FR-JUDGE-004  
**描述**：每帧逻辑更新时，系统应检测所有活跃节奏点：若某节奏点底部超出判定区域下边界50px（即 `y > judgeZone.y + judgeZone.height + 50`），自动判定为 `MISS` 并回收该节奏点。

#### 3.6.5 最优节奏点选择

**需求编号**：FR-JUDGE-005  
**描述**：当同一轨道判定区域内有多个节奏点时，系统应选取中心点距离判定线（perfectY）最近的节奏点作为命中目标。

### 3.7 评分系统

#### 3.7.1 基础分数

**需求编号**：FR-SCORE-001  
**描述**：
| 判定等级 | 基础分数 |
|----------|----------|
| PERFECT | 300分 |
| GOOD | 200分 |
| MISS | 0分 |

#### 3.7.2 连击加成

**需求编号**：FR-SCORE-002  
**描述**：每次 PERFECT 或 GOOD 命中使连击数加1，实际得分公式：
```
实际得分 = floor(基础分数 × min(1 + combo × 0.1, 3))
```
连击加成上限为3倍（即combo ≥ 20时达到上限）。MISS时连击数归零。

#### 3.7.3 最大连击记录

**需求编号**：FR-SCORE-003  
**描述**：系统应记录本局游戏中出现的最高连击数（`maxCombo`），并在游戏结算界面展示。

#### 3.7.4 精度统计

**需求编号**：FR-SCORE-004  
**描述**：精度计算公式：
```
精度 = (perfectHits + goodHits) / totalHits × 100%
```
保留一位小数，实时显示于界面右上角，游戏结算时显示最终精度。

### 3.8 反馈系统

#### 3.8.1 视觉反馈

**需求编号**：FR-FB-001  
**描述**：
- 按键按下时，对应轨道判定区域白色半透明高亮，持续10帧
- 判定区域颜色区分（金色PERFECT区域，绿色GOOD区域）
- 实时显示分数、连击、BPM、精度等信息

#### 3.8.2 音效反馈

**需求编号**：FR-FB-002  
**描述**：
| 判定等级 | 音效 |
|----------|------|
| PERFECT | 鼓声 + PERFECT音效（`audio/bullet.mp3`） |
| GOOD | 鼓声 + GOOD音效（`audio/bullet.mp3`） |
| MISS | MISS音效（`audio/boom.mp3`） |

鼓声优先使用 Web Audio API 合成；若不支持，降级为文件播放；若文件不存在，静默处理。

#### 3.8.3 触觉反馈

**需求编号**：FR-FB-003  
**描述**：
| 操作 | 震动类型 |
|------|----------|
| PERFECT命中 | 中等震动（`medium`） |
| GOOD命中 | 轻微震动（`light`） |
| 按键按下（虚拟按键） | 轻微震动（`light`） |
| 点击模式唤醒按钮 | 中等震动（`medium`） |

震动通过微信 `wx.vibrateShort` API 实现，若设备不支持则静默跳过。

### 3.9 节奏选择器

#### 3.9.1 初始界面显示

**需求编号**：FR-SEL-001  
**描述**：游戏启动时，系统应显示节奏选择器（`initial`模式），覆盖整个游戏画面，玩家在此配置游戏参数。

#### 3.9.2 节奏模式选择

**需求编号**：FR-SEL-002  
**描述**：节奏选择器应提供下拉列表，展示从 `rhythm-patterns.json` 加载的所有可用节奏模式，包含：
- 模式名称（如"基础4拍节奏"）
- 难度标签（easy / medium / hard / expert）
- BPM信息

#### 3.9.3 BPM设置

**需求编号**：FR-SEL-003  
**描述**：玩家应能通过以下方式设置BPM：
- **按钮调节**：点击"+"或"-"按钮，以5为步长调整BPM
- **键盘输入**：双击BPM数字区域，弹出虚拟键盘，直接输入数值
- **BPM范围**：40~200，超出范围时截断至边界值

#### 3.9.4 虚拟键盘

**需求编号**：FR-SEL-004

## 5. 非功能性需求

### 5.1 性能需求

| 指标 | 要求 |
|------|------|
| 目标帧率 | 60 FPS |
| 最大帧间隔 | ≤ 16.7ms |
| 输入响应延迟 | ≤ 50ms |
| 判定精度 | ±16ms（一帧范围内） |
| 节拍生成延迟 | ≤ 16ms |

### 5.2 内存管理

- 使用对象池（Pool）管理节奏点对象，避免频繁内存分配和垃圾回收
- 对象池可复用已回收的节奏点对象，减少GC压力
- 动画对象在播放完成后及时从列表清除

### 5.3 稳定性需求

- 游戏运行期间不得发生崩溃或异常退出
- 所有外部资源（JSON文件、音频文件）加载失败时，系统应有完整的降级方案，保证游戏基本功能可用
- 所有关键操作应有 try-catch 错误处理和日志输出

### 5.4 可维护性需求

- 代码采用 ES6+ 模块化组织，单一职责原则
- 每个类/方法应有 JSDoc 注释说明功能、参数和返回值
- 游戏常量（屏幕尺寸、判定参数等）应集中定义，便于调整

### 5.5 可扩展性需求

- 节奏模式通过外部 JSON 文件配置，新增模式无需修改代码
- 音符类型、判定区域参数应易于扩展和调整
- 评分公式和连击加成逻辑应集中在 `DataBus` 中，便于修改
